## 1. 项目说明

### 1.1 核心定位

基于Claude Code和多智能体协作的餐饮行业数智化平台。8大业务组、67个专业智能体协同工作，覆盖战略规划、品牌营销、数据分析、运营中台、供应链管理、门店筹建、系统开发全链路。

**技术栈**: Claude Code + Sonnet 4.5 + MCP生态

### 1.2 业务矩阵

| 业务组 | 系列 | 数量 | 核心职能 | 组长 |
|--------|------|------|----------|------|
| 战略组 | G系列 | 9个 | 战略规划、经营分析、产品力打造、精细化管理 | GG |
| 创意组 | X系列 | 9个 | 广告策划、文案创作、平面设计、短视频制作 | XX |
| 情报组 | E系列 | 8个 | 公开调研、网站采集、深度分析、数据管理 | EE |
| 行政组 | R系列 | 9个 | 财务、人事、法务、秘书、飞书、文件、COS存储 | RR |
| 中台组 | M系列 | 9个 | 美团运营、营销管理、供应链、报表、小程序 | MM |
| 供应组 | S系列 | 7个 | 供应需求分析、采购管理、库存控制、成本管理、供应商关系 | SS |
| 筹建组 | Z系列 | 6个 | 需求分析、平面图设计、空间设计、BIM建模、动画渲染 | ZZ |
| 开发组 | F系列 | 11个 | 产品管理、前后端开发、数据库、API、测试部署 | FF |

### 1.3 开发规范

**物料管理**:

| 类型 | 位置 | 命名规范 |

## 3. Agents智能体

### 智能体总览

**67个专业智能体** 分布于8大业务组：

```
.claude/agents/
├── 战略组/  # G0-GG (9个) - 战略规划、经营分析、产品力、精细化管理
├── 创意组/  # X0-XX (9个) - 广告策划、文案设计、短视频制作
├── 情报组/  # E0-EE (8个) - 公开调研、数据采集、深度分析、云存储
├── 行政组/  # R0-RR (9个) - 财务人事法务、秘书飞书、文件COS存储
├── 中台组/  # M0-MM (9个) - 美团运营、营销管理、供应链、成本卡
├── 供应组/  # S0-SS (7个) - 供应需求分析、采购管理、库存控制、成本管理、供应商关系
├── 筹建组/  # Z0-ZZ (6个) - 需求分析、平面图设计、空间设计、BIM建模、动画渲染
└── 开发组/  # F0-FF (11个) - 全栈开发、数据库、API、测试部署
```

**典型工作流**: `需求分析(X0) → 组长调度(XX) → 专业执行(X1-X7) → 质量整合(XX) → 成果交付`

### M系列特别说明

> ⚠️ **重要**: M1/M2/M3提供业务知识，M4提供真实执行

**正确使用流程**:
1. 读取M1/M2/M3配置 → 获取业务知识
2. 选择执行方式: chrome-mcp直接操作 或 M4智能体自动化
3. 真实执行 → 调用工具操作页面（前提：用户已登录）
4. 验证结果 → 生成基于真实执行的报告

## 7. 意图图谱

### 7.1 意图识别原则

当用户提出需求时，系统通过**关键词匹配** → **任务类型识别** → **智能体路由** → **专业执行**完成任务。

```
意图分析流程:
  1. 提取业务关键词
  2. 判断任务类型（战略/创意/情报/行政/运营/筹建/开发）
  3. 调用对应组长智能体
  4. 组长分解任务并调度专业智能体
```

## 10. 错误与纠正记录

> 基于MANUS原则的错误记录与适应性优化系统
> 通过系统化记录错误、恢复过程和改进措施，实现持续学习和能力提升

### 10.1 记录原则

```yaml
MANUS技巧:
  M - Mistake: 清晰定义错误
  A - Acknowledgment: 承认错误
  N - New Understanding: 建立新认知
  U - Updated Approach: 更新方法
  S - Systematic Prevention: 系统性预防

触发机制:
  - 自动触发: Claude检测到错误时主动记录
  - 手动触发: 用户执行 `/X` 命令记录错误

记录标准:
  ✅ 影响用户体验的错误
  ✅ 重复发生的错误模式
  ✅ 需要更新认知模型的错误
  ✅ 可能影响其他开发者的错误
```

### 10.2 错误分类

```yaml
错误类型标签:
  - LOGIC: 逻辑错误
  - SYNTAX: 语法错误
  - PERMISSION: 权限错误
  - TOOL_USE: 工具使用错误
  - FILE_OP: 文件操作错误
  - INTEGRATION: 集成错误
  - PERFORMANCE: 性能问题

严重级别:
  - HIGH: 严重影响功能或用户信任
  - MEDIUM: 影响体验但可恢复
  - LOW: 轻微问题，易于修复
```

### 10.3 结构化错误数据系统

**数据存储位置**: `learning/errors/`

```yaml
核心文件:
  ERRORS.jsonl:
    格式: 行分隔JSON（每行一个完整错误记录）
    写入: /X (MANUS命令)
    读取: /learn (分析和优化)
    特点: 完整的MANUS五步法记录

  PATTERNS.json:
    格式: 结构化JSON
    内容: 提炼的错误模式库
    更新: /learn自动分析生成
    特点: 反模式识别和预防规则

  STATS.json:
    格式: 结构化JSON
    内容: 错误统计和趋势分析
    更新: /learn自动计算
    特点: 支持优化决策

  exports/:
    格式: Markdown报告
    内容: 人类可读的分析报告
    生成: /learn导出
    特点: 可视化图表和洞察
```

**闭环工作流**:

```
错误发生
  → /X记录到ERRORS.jsonl (完整MANUS数据)
  → /learn读取和分析
  → 更新PATTERNS.json (模式提炼)
  → 更新STATS.json (统计趋势)
  → 生成OPTIMIZATION.md (优化建议)
  → 系统改进和能力提升
```
